# Code Change Docs

## Use This When

- the implementation spans many files
- the mechanic crosses shared/server/client/resources
- reviewers will need a short explanation of the architecture

## Recommended Sections

- Problem
- Ownership and file placement
- Runtime flow
- Prediction or networking notes
- Validation

## Rules

- Keep notes brief and technical.
- Prefer stable paths and type names.
- Do not create permanent markdown files unless the task calls for repo docs.
